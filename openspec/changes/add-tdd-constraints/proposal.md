## Why

确保开发过程中遵循 TDD 原则，保证代码质量，减少 bug 引入，并形成可持续的测试习惯。

**约束目标：**
- 强制 TDD 开发流程
- 保障单元测试覆盖率
- 提升代码可维护性

## What Changes

**新增约束文件：**
1. `.claude/skills/tdd-constraints.md` - TDD 开发约束文档

**约束内容：**

| 约束项 | 要求 | 说明 |
|--------|------|------|
| TDD 开发流程 | 必须遵循 Red-Green-Refactor | 先写测试，再实现 |
| 单元测试覆盖率 | ≥ 60% | 可使用 JaCoCo/Jest 等工具检查 |
| 测试命名 | 描述性命名 | 说明测试的行为而非实现 |
| 测试结构 | AAA 模式 | Arrange-Act-Assert |

## Why TDD

| 收益 | 说明 |
|------|------|
| 减少调试时间 | 测试即文档，快速定位问题 |
| 提升代码质量 | 迫使开发者思考接口设计 |
| 防止回归 | 每次修改都有测试保障 |
| 提升信心 | 放心重构 |

## Why 60%

- MVP 阶段资源有限，80% 可能过于严格
- 60% 覆盖可捕获大部分关键路径
- 后续可逐步提升到 80%

## Capabilities

### 建议的 TDD 流程

```
1. RED: 写一个失败的测试（描述期望行为）
2. GREEN: 写最少的代码让测试通过
3. REFACTOR: 重构代码，保持测试通过
4. REPEAT: 重复以上步骤
```

### 建议的测试覆盖率门槛

| 项目阶段 | 覆盖率要求 |
|----------|------------|
| MVP 开发 | ≥ 60% |
| 成熟项目 | ≥ 80% |
| 关键模块 | ≥ 90% |

## 其他建议

| 建议 | 原因 |
|------|------|
| 测试独立 | 每个测试独立运行，不依赖顺序 |
| 单一职责 | 每个测试只验证一个行为 |
| 快速反馈 | 测试应在秒级完成 |
| Mock 外部依赖 | 单元测试不应依赖外部系统 |
