## Why

企业内部多个业务系统需要在关键业务事件发生时调用外部供应商的 HTTP(S) API 进行通知（如广告注册回调、CRM 状态变更、库存变更等）。当前各业务系统直接调用外部 API 存在以下问题：

- **耦合严重**：业务系统需了解每个外部 API 的地址、认证方式、请求格式
- **可靠性无保障**：网络抖动、外部服务临时不可用时，通知直接丢失
- **缺乏统一监控**：无法统一追踪通知投递状态、失败重试、延迟等
- **缺乏管理界面**：配置和监控依赖数据库直连或代码修改

需要构建一个统一的 API 通知服务，作为企业内所有业务系统与外部 API 之间的桥梁。

## What Changes

1. **新增通知服务（单体架构）**：前后端分离的 Web 应用
   - **后端 API (SpringBoot)**：接收通知请求、管理配置、投递通知
   - **前端 Web (Vue 3)**：配置管理界面
2. **统一接入协议**：业务系统只需按统一格式投递通知，无需关心外部 API 细节
3. **可靠投递机制**：支持同步/异步投递、失败重试
4. **配置化管理**：外部 API 的地址、认证信息、请求模板集中管理
5. **状态记录**：记录投递状态（成功/失败），支持后续排查

## Capabilities

### New Capabilities

- `notification-api`: 接收业务系统的通知请求，路由到投递服务
- `notification-sender`: 将通知实际投递到外部 API，处理认证、重试逻辑
- `notification-config`: 外部 API 配置管理（地址、认证、Header/Body 模板、超时设置）
- `notification-frontend`: Web UI（配置管理页面）

### Modified Capabilities

（无 - 这是全新构建的系统）

## Impact

- **新增应用**：单体架构的通知服务（前后端分离部署）
- **前端**：Vue 3 + Vite SPA
- **后端**：SpringBoot RESTful API
- **数据库**：H2（开发/测试）
- **外部依赖**：各业务系统改为向通知服务投递消息

## 演进备选方案（暂不实现）

| 触发条件 | 引入方案 |
|----------|----------|
| 通知量 > 1000/秒 | 引入 RabbitMQ 消息队列 |
| 失败通知需人工处理 | 引入死信队列 |
| 需要限流保护 | 引入限流机制（令牌桶/滑动窗口） |
| 需要可视化监控 | 引入 Prometheus + Grafana 监控面板 |
| 复杂度显著增加 | 拆分为微服务 |
