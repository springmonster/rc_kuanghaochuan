## Why

企业内部多个业务系统需要在关键业务事件发生时调用外部供应商的 HTTP(S) API 进行通知（如广告注册回调、CRM 状态变更、库存变更等）。当前各业务系统直接调用外部 API 存在以下问题：

- **耦合严重**：业务系统需了解每个外部 API 的地址、认证方式、请求格式
- **可靠性无保障**：网络抖动、外部服务临时不可用时，通知直接丢失
- **缺乏统一监控**：无法统一追踪通知投递状态、失败重试、延迟等

需要构建一个统一的 API 通知服务，作为企业内所有业务系统与外部 API 之间的桥梁。

## What Changes

1. **新增 API 通知服务**：一个专门负责将通知投递到外部 HTTP(S) API 的中间层服务
2. **统一接入协议**：业务系统只需按统一格式投递通知，无需关心外部 API 细节
3. **可靠投递机制**：支持同步/异步投递、失败重试
4. **统一监控面板**：追踪投递成功率、延迟、失败原因
5. **配置化管理**：外部 API 的地址、认证信息、请求模板集中管理

## Capabilities

### New Capabilities

- `notification-api`: 负责接收业务系统的通知请求，根据配置路由到对应的外部 API
- `notification-sender`: 负责将通知实际投递到外部 API，处理认证、重试逻辑
- `notification-config`: 外部 API 的配置管理（地址、认证、Header/Body 模板、超时设置）
- `notification-monitor`: 投递状态的追踪、指标采集、告警

### Modified Capabilities

（无 - 这是全新构建的系统）

## Impact

- **新增服务**：部署一个新的微服务（Notification Service）
- **依赖组件**：轻量级依赖，主要使用标准 HTTP 客户端库
- **外部依赖**：各业务系统需改造，将原有的直接 HTTP 调用改为向通知服务投递消息
- **监控对接**：与现有监控系统（如 Prometheus/Grafana）集成

## 演进备选方案（暂不实现）

以下为未来流量激增或复杂度显著增长时可引入的方案：

- **消息队列**（RabbitMQ/RocketMQ）：当通知量级达到数千/秒时，引入队列解耦
- **死信队列**：当重试机制无法满足可靠性需求时，引入人工处理失败通知
