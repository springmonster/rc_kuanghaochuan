## Context

企业内部多个业务系统（微服务）需要在关键业务事件发生时通知外部第三方系统。当前各业务系统直接调用外部 API 存在高度耦合、可靠性无保障、监控分散等问题。

**当前痛点：**
- 业务系统需直接维护每个外部 API 的调用代码，包括认证、请求格式等
- 网络抖动或外部服务不可用时，通知直接丢失
- 各系统独立实现，缺乏统一的投递状态追踪

**关键约束：**
- 发送方（内部微服务）：技术栈不统一，直接访问外部 URL 能力不一致
- 接收方（外部 API）：不可控，可能无多环境，HTTP 存在中间人攻击风险
- 业务系统不关心外部 API 返回值，只关心通知是否送达

## Goals / Non-Goals

**Goals:**
- 提供统一的 API 通知接入层，业务系统只需按标准格式投递
- 保证通知至少投递一次（at-least-once）
- 支持异步投递，调用方无需等待外部 API 响应
- 提供完整的投递状态追踪和失败告警
- 集中管理外部 API 配置（地址、认证、超时、重试策略）

**Non-Goals:**
- 不处理外部 API 的返回值（业务系统不关心 response body）
- 不实现复杂的消息路由（固定 1:1 映射，通知 → 目标 API）
- 不支持同步调用场景（所有投递均为异步）
- 不处理外部 API 的回调（单向通知）

## Decisions

### Decision 1: 架构模式 - 选型消息队列 vs 直连 HTTP

**选择：消息队列（RabbitMQ/RocketMQ）**

**理由：**
- 业务系统与外部 API 完全解耦，发送方不阻塞
- 天然支持失败重试和死信队列
- 流量削峰，保护外部 API 不过载
- 投递状态清晰可追踪

**替代方案考虑：**
- 直连 HTTP：简单但可靠性差，无重试缓冲
- 数据库 + 轮询：延迟高，资源消耗大

### Decision 2: 消息投递语义 - At-Least-Once

**选择：At-Least-Once（至少一次）**

**理由：**
- 外部 API 通知通常为幂等操作（如更新 CRM 状态）
- 业务优先保证不丢消息，接受少量重复
- 实现复杂度适中，重试机制成熟

**取舍：**
- 可能产生重复通知，需要外部系统做好幂等处理
- 不能保证 exactly-once 的情况下，接受重复是合理权衡

### Decision 3: 重试策略 - 指数退避 + 最大次数

**选择：指数退避，最大 5 次重试**

**理由：**
- 指数退避避免对暂时性故障的外部 API 造成压力
- 5 次重试覆盖大多数临时故障场景（网络抖动、目标服务重启）
- 超过 5 次的失败进入死信队列，等待人工介入

**参数建议：**
- 基础延迟：1s
- 退避倍数：2
- 最大延迟：60s
- 最大重试：5次

### Decision 4: 认证方式 - 配置化支持多种认证

**选择：配置中心统一管理认证信息**

**理由：**
- 不同外部 API 认证方式不同（API Key、Bearer Token、Basic Auth、OAuth2）
- 认证信息敏感，需加密存储
- 业务系统不应接触认证细节，只知道"发送通知"

**实现：**
- Destination Config 中存储认证信息（加密）
- Dispatcher 根据配置自动添加认证 header
- 不在代码中硬编码认证信息

### Decision 5: HTTPS 安全 - 跳过证书验证（可配置）

**选择：默认验证 HTTPS 证书，可配置跳过**

**理由：**
- HTTPS 是基本原则，应该验证证书
- 但部分开发/测试环境使用自签名证书
- 提供可配置的 skip_cert_verify 选项，仅在明确风险的情况下使用
- 生产环境必须验证证书

### Decision 6: 监控方案 - Prometheus + Grafana

**选择：基于 Prometheus metrics 暴露指标**

**理由：**
- 与现有监控系统兼容
- 指标标准化（notifications_sent/delivered/failed/duration）
- 支持多维度标签（destination、source_system）
- 告警规则标准化

## Risks / Trade-offs

**[Risk] 外部 API 长期不可用 → 通知积压**
→ **Mitigation**: 监控队列深度，设置积压告警阈值；外部 API 恢复后自动消化积压

**[Risk] 消息队列单点故障**
→ **Mitigation**: RabbitMQ 集群模式 + 持久化 + 镜像队列

**[Risk] 认证信息泄露**
→ **Mitigation**: 配置加密存储（如 Vault），最小权限原则，审计日志

**[Risk] 重复通知导致业务问题**
→ **Mitigation**: 在通知中包含幂等 ID；要求外部系统做好幂等处理

**[Trade-off] 异步投递导致业务反馈延迟**
→ 业务系统需接受通知发出后可能有延迟，无法实时获得投递结果

## Migration Plan

**Phase 1: 搭建基础框架**
1. 部署 RabbitMQ 集群（开发环境）
2. 实现 Router 基本功能
3. 实现 Dispatcher 基本投递功能

**Phase 2: 配置管理**
4. 实现 Config 管理界面
5. 接入第一个外部 API 进行联调

**Phase 3: 监控与告警**
6. 接入 Prometheus metrics
7. 配置 Grafana 看板
8. 配置失败告警

**Phase 4: 生产就绪**
9. 迁移 RabbitMQ 到生产集群
10. 完善 Config 加密存储
11. 业务系统逐步接入

**回滚策略：**
- 新通知可通过配置切换回直连模式
- 队列中积压的消息在回滚后继续处理

## Open Questions

1. **消息持久化策略**：是否需要所有消息持久化？内存队列 vs 磁盘队列的权衡？
2. **外部 API 超时设置**：不同 API 可能需要不同超时，是统一默认值还是完全可配置？
3. **通知顺序性**：是否需要保证同一 destination 的通知顺序？如不需要，可进一步优化吞吐。
4. **监控数据保留**：监控指标和投递日志保留多长时间？
5. **多环境支持**：是否需要区分 dev/staging/prod 的 destination 配置？
