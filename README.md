# API通知系统设计与实现

## 问题理解

1. 需求描述

```
企业内部多个业务系统在关键事件发⽣时，需要调⽤外部系统供应商提供的 HTTP(S) API进⾏通知。例如：

- ⽤户通过第三⽅⼴告系统引流并成功注册后，通知对应的⼴告系统
- ⽤户订阅付款成功后，通知 CRM 系统更改 Contact 状态
- ⽤户购买商品后，通知库存系统进⾏库存变更

不同供应商的 API：
- 请求地址不同
- Header / Body 格式不同

业务系统本⾝：
- 不需要关⼼外部 API 的返回值
- 只需确保通知请求能够被稳定、可靠地送达

```
2. 我的理解

- API系统
  - 这是一个调用三方服务时，可靠消息的投递问题
  - 不需要关心外部API的返回值，应该是特指response body
  - 需要reponse status code来确定是否送达
  - 要保证可靠性
  - 要保证从应用层级，系统层级，基础设施层级做到全方位的监控
- 发送方：企业内部多个业务系统（可以认为是微服务，完全可控）
  - 发送方可能使用的技术栈不统一
  - 发送方是否可以直接访问接收方的URL
  - 接收方的认证方式有很多
  - 接收方如果使用HTTPS，证书过期如何处理
- 接收方：外部系统供应商提供的服务（不可控）
  - 接收方可能没有多套环境匹配
  - 如果使用HTTP，可能存在中间者攻击

## 系统边界
### 哪些问题你选择在这个系统中解决？

- 统一接收业务系统的通知请求
- 统一管理外部 API 的配置（地址、认证、超时，重试策略）
- 统一处理投递失败的重试逻辑
- 记录投递状态（成功/失败），支持后续排查

### 哪些问题你明确选择不解决？为什么？

- **外部 API 返回值处理**：业务系统明确不需要关心返回值
- **外部 API 的回调**：当前需求为单向通知，暂不处理回调
- **复杂消息路由**：当前只有 1:1 的通知映射，暂不需要复杂路由
- **限流**：作为后续优化项，第一版聚焦核心投递功能
- **监控面板**：作为后续优化项，第一版先记录状态到数据库

## 可靠性与失败处理
### 你选择的通知投递语义是什么？（例如：⾄少⼀次、其他）

**至少一次（At-Least-Once）**

- 外部 API 通知通常为幂等操作
- 优先保证不丢消息，接受少量重复

### 在外部系统失败或⻓期不可⽤的情况下，你的处理策略是什么？

**重试策略：指数退避**
- 基础延迟：1s
- 退避倍数：2
- 最大延迟：60s
- 最大重试：5次

**重试耗尽后：**
- 记录失败状态和原因到数据库
- 支持通过数据库查询失败记录
- 暂不引入死信队列（未来可选项）

## 取舍与演进
### 在 AI 的⽅案建议中，你认为哪些设计是"过度的"，并选择不采纳？

| 过度设计 | 不采纳原因 |
|----------|------------|
| **消息队列（RabbitMQ/RocketMQ）** | 当前通知量级未知，引入消息队列增加运维复杂度和部署成本。第一版优先保证简单，消息队列作为未来流量激增（>1000/秒）时的可选项 |
| **死信队列（Dead Letter Queue）** | 当前重试机制（5次指数退避）足够应对大多数临时故障。死信队列作为未来需要人工处理失败通知时的可选项 |
| **限流机制** | 第一版量级有限，聚焦核心投递功能。限流作为后续优化项 |
| **监控面板（Prometheus + Grafana）** | 第一版先记录状态到数据库，可通过数据库查询。监控面板作为后续优化项 |

**为什么不一开始就用 RabbitMQ？**
- 当前业务场景的通知量级未知，可能只需要几百/天
- RabbitMQ 引入需要额外的运维、监控、集群配置
- 第一版聚焦核心投递能力，避免过度工程化
- 如果未来流量激增或复杂度增加，可以在不停机的情况下平滑演进

### 如果这是⼀个第⼆版系统，未来流量或复杂度显著增⻓时，你会如何演进？

**演进触发条件和方案：**

| 触发条件 | 引入方案 |
|----------|----------|
| 通知量 > 1000/秒 | 引入 RabbitMQ 消息队列，实现异步投递和削峰 |
| 需要保证消息不丢失 | 引入消息持久化 + 确认机制（publisher confirms） |
| 失败通知需要人工处理 | 引入死信队列 + 管理界面 |
| 需要限流保护 | 引入限流机制（令牌桶/滑动窗口） |
| 需要可视化监控 | 引入 Prometheus + Grafana 监控面板 |
| 需要跨多个服务协调 | 引入分布式事务或 saga 模式 |

### 技术栈选择

| 组件 | 技术选型 | 原因 |
|------|----------|------|
| 前端 | Vue 3 + Vite | 主流框架，生态成熟，组件化开发效率高 |
| 后端 | SpringBoot | 企业级成熟框架，生态完善，团队熟悉 |
| 数据库 | H2 | 零配置嵌入式数据库，开发测试方便，未来可迁移 |
